# Database Naming Conventions

This document defines the database schema design rules for the **MAD ERP Backend** project. Follow these conventions when creating new tables, fields, and database structures.

## ⚠️ Critical Rule: organization_id Requirement

**IMPORTANT**: ALL tables MUST have `organization_id` field for tenant isolation, **EXCEPT**:
- ✅ **Exceptions (NO organization_id)**:
  - `enum_*` tables (static global enumerations)
  - `info_*` tables (global reference data)
  - `sys_*` tables in `adm` schema that are truly global (e.g., `sys_user`, `sys_role`, `sys_table`)

- ✅ **Required (MUST have organization_id)**:
  - `hl_*` tables (tenant-specific reference data)
  - `doc_*` tables (business documents)
  - Domain-specific tables in schemas like `hrm`, `org`, `std`, `fin`, `com`, `my`, `app`

**Rule of Thumb**: If the table is in a domain schema (like `hrm`, `org`, `std`), it MUST have `organization_id` for tenant isolation.

## Table Prefix Rules

MAD ERP uses 5 table prefix types to categorize database tables by their purpose and behavior:

### 1. `enum_` - Static Enumerations

**Purpose:** Static lookup tables with fixed IDs known at development time.

**Characteristics:**
- Primary key is **NOT** auto-incrementing (manually assigned)
- IDs can be used directly in source code constants
- Has `state_id` field (except for foundational enums like `enum_state` itself)
- **NO** `created_by` or `last_modified_by` fields (only timestamps)

**Examples:** `enum_state`, `enum_status`, `enum_gender`, `enum_language`, `enum_app`

**SQL Example:**
```sql
CREATE TABLE IF NOT EXISTS adm.enum_state (
    id                      INTEGER                     NOT NULL,
    order_code              VARCHAR(4),
    code                    VARCHAR(20)                 NOT NULL,
    short_name              VARCHAR(50)                 NOT NULL,
    full_name               VARCHAR(100)                NOT NULL,
    created_at              TIMESTAMP WITHOUT TIME ZONE NOT NULL DEFAULT NOW(),
    last_modified_at        TIMESTAMP WITHOUT TIME ZONE,
    CONSTRAINT pk_adm_enum_state PRIMARY KEY (id)
);
```

### 2. `info_` - Global Reference Data

**Purpose:** Global master data shared across all organizations.

**Characteristics:**
- Has `state_id` field (Active/Passive)
- **NO** `organization_id` field (not tenant-specific)
- Has user audit fields (`created_by`, `last_modified_by`)
- Primary key auto-generated: `INTEGER GENERATED BY DEFAULT AS IDENTITY`

**Examples:** `info_country`, `info_currency`, `info_organization`, `info_bank`, `info_region`, `info_district`

**SQL Example:**
```sql
CREATE TABLE IF NOT EXISTS adm.info_country (
    id                      INTEGER GENERATED BY DEFAULT AS IDENTITY,

    -- Required info_ fields
    order_code              VARCHAR(4),
    code                    VARCHAR(50)                 NOT NULL,
    short_name              VARCHAR(250)                NOT NULL,
    full_name               VARCHAR(500)                NOT NULL,
    state_id                INTEGER                     NOT NULL,

    -- Audit fields
    created_at              TIMESTAMP WITHOUT TIME ZONE NOT NULL DEFAULT NOW(),
    created_by              INTEGER,
    last_modified_at        TIMESTAMP WITHOUT TIME ZONE,
    last_modified_by        INTEGER,

    CONSTRAINT pk_adm_info_country PRIMARY KEY (id),
    CONSTRAINT fk_adm_info_country_state_id FOREIGN KEY (state_id) REFERENCES adm.enum_state (id)
);
```

### 3. `hl_` - Tenant-Specific Reference Data

**Purpose:** Reference data specific to each organization (tenant). "HL" stands for "Header Line" or tenant-specific master data.

**Characteristics:**
- Has `state_id` field
- **ALWAYS** has `organization_id` field (tenant-isolated)
- Organization administrators can edit their own records
- Has user audit fields
- Primary key typically auto-generated

**Examples:** `hl_person`, `hl_employee`, `hl_position`, `hl_department`

**SQL Example:**
```sql
CREATE TABLE IF NOT EXISTS hrm.hl_employee (
    id                      INTEGER GENERATED BY DEFAULT AS IDENTITY,

    -- Business-specific fields
    person_id               INTEGER                     NOT NULL,
    short_name              VARCHAR(250)                NOT NULL,
    full_name               VARCHAR(500)                NOT NULL,

    -- Required hl_ fields
    state_id                INTEGER                     NOT NULL,
    organization_id         INTEGER                     NOT NULL,

    -- Audit fields
    created_at              TIMESTAMP WITHOUT TIME ZONE NOT NULL DEFAULT NOW(),
    created_by              INTEGER,
    last_modified_at        TIMESTAMP WITHOUT TIME ZONE,
    last_modified_by        INTEGER,

    CONSTRAINT pk_hrm_hl_employee PRIMARY KEY (id),
    CONSTRAINT fk_hrm_hl_employee_state_id FOREIGN KEY (state_id) REFERENCES adm.enum_state (id),
    CONSTRAINT fk_hrm_hl_employee_organization_id FOREIGN KEY (organization_id) REFERENCES adm.info_organization (id)
);
```

### 4. `doc_` - Business Documents

**Purpose:** Business process transactions and documents with lifecycle management.

**Characteristics:**
- **ALWAYS** has: `status_id`, `organization_id`, `doc_on`, `doc_number`, `table_id`
- Represents business documents with status workflow
- Has user audit fields
- Primary key typically `BIGINT GENERATED ALWAYS AS IDENTITY` (for high-volume transactional data)
- **NO** `state_id` (uses `status_id` instead)

**Examples:** `doc_cadastre`, `doc_structure_passport`, `doc_application`, `doc_appointment_employee`

**SQL Example:**
```sql
CREATE TABLE org.doc_cadastre (
    id                  BIGINT GENERATED ALWAYS AS IDENTITY,
    doc_number          VARCHAR(30)                 NOT NULL,
    doc_on              DATE                        NOT NULL,

    -- Business-specific fields
    full_name           VARCHAR(250)                NOT NULL,
    cadastre_number     VARCHAR(50)                 NOT NULL,
    total_area          NUMERIC(18,2)               NOT NULL DEFAULT 0,

    -- Required doc_ fields
    status_id           INT                         NOT NULL,
    table_id            INT                         NOT NULL,
    organization_id     INT                         NOT NULL,

    -- Audit fields
    created_at          TIMESTAMP WITHOUT TIME ZONE NOT NULL DEFAULT NOW(),
    created_by          BIGINT,
    last_modified_at    TIMESTAMP WITHOUT TIME ZONE,
    last_modified_by    BIGINT,

    CONSTRAINT pk_org__doc_cadastre PRIMARY KEY (id),
    CONSTRAINT uc_org__doc_cadastre__organization_id__doc_number UNIQUE (organization_id, doc_number),
    CONSTRAINT fk_org__doc_cadastre__status_id FOREIGN KEY (status_id) REFERENCES adm.enum_status (id),
    CONSTRAINT fk_org__doc_cadastre__organization_id FOREIGN KEY (organization_id) REFERENCES adm.info_organization (id),
    CONSTRAINT fk_org__doc_cadastre__table_id FOREIGN KEY (table_id) REFERENCES adm.sys_table (id)
);
```

### 5. `sys_` - System Infrastructure Tables

**Purpose:** Core system functionality and infrastructure.

**Characteristics:**
- System tables for users, roles, permissions, logs, configurations
- **Context-dependent** `organization_id` requirement:
  - Global system tables (in `adm` schema): NO `organization_id` (e.g., `sys_user`, `sys_role`, `sys_table`)
  - Domain-specific system tables: MUST have `organization_id`
- May or may not have `state_id` (varies by table)
- Has user audit fields

**Examples (adm schema - NO organization_id):** `sys_user`, `sys_role`, `sys_permission`, `sys_table`

**Examples (domain schemas - HAS organization_id):** Domain-specific system configurations, logs, settings

#### Special Pattern: `sys_*_manage` Tables

**Purpose:** Management/tracking tables that record the lifecycle and changes of entities based on documents.

**Naming Pattern:** `sys_{entity}_manage`

**Key Characteristics:**
- Track entity states over time (start_on, end_on)
- Reference source documents via `table_id` + `doc_id` (or `document_id`)
- Track termination/end documents via `end_table_id` + `end_doc_id` (or `end_document_id`)
- **ALWAYS** have `organization_id` (tenant-specific)
- **NO** `state_id` or `status_id` (track validity via date ranges)
- Use `BIGINT` for primary key (high-volume transactional data)
- Primary key typically named `{entity}_manage_id`
- **Required CHECK constraints:**
  - `CHECK (start_on < end_on)` - Ensures start date is before end date
  - `CHECK (close_on <= end_on)` - If close_on field exists, must be before or equal to end_on

**Common Fields:**
```sql
id / {entity}_manage_id  BIGINT GENERATED ALWAYS AS IDENTITY,
table_id                 INTEGER NOT NULL,           -- Source document type
doc_id / document_id     BIGINT NOT NULL,            -- Source document ID
start_on                 DATE NOT NULL,              -- Effective start date
end_on                   DATE,                       -- Effective end date (NULL = active)
end_table_id             INTEGER,                    -- Termination document type
end_doc_id / end_document_id  BIGINT,                -- Termination document ID
close_on                 DATE,                       -- Close/completion date (optional)
close_reason             VARCHAR(600),               -- Close reason description (optional)
close_table_id           INTEGER,                    -- Close document type (optional)
close_doc_id             BIGINT,                     -- Close document ID (optional)
organization_id          INTEGER NOT NULL,
message                  VARCHAR(600),               -- Additional message/note (optional)
created_at               TIMESTAMP WITHOUT TIME ZONE NOT NULL,
created_by               INTEGER/BIGINT,
last_modified_at         TIMESTAMP WITHOUT TIME ZONE,
last_modified_by         INTEGER/BIGINT,
is_deleted               BOOLEAN DEFAULT FALSE       -- Soft delete flag (optional)
```

**Examples:**
- `hrm.sys_employee_manage` - Tracks employee appointments, transfers, dismissals
- `std.sys_student_manage` - Tracks student enrollment, grade changes, withdrawals
- `std.sys_music_student_manage` - Tracks music school student status

**SQL Example:**
```sql
CREATE TABLE hrm.sys_employee_manage (
    id                       BIGINT GENERATED ALWAYS AS IDENTITY,

    -- Required doc_ fields
    doc_table_id             INTEGER                     NOT NULL,
    doc_id                   BIGINT                      NOT NULL,
    start_on                 DATE                        NOT NULL,
    end_on                   DATE,
    end_table_id             INTEGER,
    end_doc_id               BIGINT,
    close_on                 DATE,
    close_reason             VARCHAR(600),
    close_table_id           INTEGER,
    close_doc_id             BIGINT,
    organization_id          INTEGER                     NOT NULL,
    is_deleted               BOOLEAN                     NOT NULL DEFAULT FALSE,

    -- Business-specific fields
    employee_id              INTEGER                     NOT NULL,
    department_id            INTEGER                     NOT NULL,
    position_id              INTEGER                     NOT NULL,
    employment_type_id       INTEGER                     NOT NULL,
    employment_rate          NUMERIC(18,3)               NOT NULL DEFAULT 0,

    -- Optional fields
    message                  VARCHAR(600),

    -- Audit fields
    created_at               TIMESTAMP WITHOUT TIME ZONE NOT NULL DEFAULT NOW(),
    created_by               BIGINT,
    last_modified_at         TIMESTAMP WITHOUT TIME ZONE,
    last_modified_by         BIGINT,

    CONSTRAINT pk_sys_employee_manage PRIMARY KEY (id),
    CONSTRAINT ch_sys_employee_manage__start_end
        CHECK (start_on < end_on),
    CONSTRAINT ch_sys_employee_manage__close_end
        CHECK (close_on <= end_on),
    CONSTRAINT fk_sys_employee_manage__doc_table
        FOREIGN KEY (doc_table_id) REFERENCES adm.sys_table(id),
    CONSTRAINT fk_sys_employee_manage__end_table_id
        FOREIGN KEY (end_table_id) REFERENCES adm.sys_table(id),
    CONSTRAINT fk_sys_employee_manage__close_table_id
        FOREIGN KEY (close_table_id) REFERENCES adm.sys_table(id),
    CONSTRAINT fk_sys_employee_manage__employee
        FOREIGN KEY (employee_id) REFERENCES hrm.hl_employee(id),
    CONSTRAINT fk_sys_employee_manage__organization
        FOREIGN KEY (organization_id) REFERENCES adm.info_organization(id)
);

-- Indexes for performance
CREATE INDEX ix_sys_employee_manage__org_id_start_on_end_on
    ON hrm.sys_employee_manage (organization_id, start_on, end_on);

CREATE INDEX ix_sys_employee_manage__doc_table_id_doc_id
    ON hrm.sys_employee_manage (doc_table_id, doc_id);
```

**Usage Pattern:**
1. When a `doc_` table creates or modifies an entity (e.g., appointment order), a record is inserted
2. `table_id` + `doc_id` reference the source document
3. `start_on` indicates when this record becomes effective
4. `end_on = NULL` means the record is currently active
5. When terminated (e.g., dismissal order), `end_on`, `end_table_id`, `end_doc_id` are set
6. Query active records: `WHERE end_on IS NULL OR end_on > CURRENT_DATE`

## Standard Field Requirements

### Fields for ALL Tables (except enum_)

```sql
id                  INTEGER/BIGINT (GENERATED),
created_at          TIMESTAMP WITHOUT TIME ZONE NOT NULL DEFAULT NOW(),
created_by          INTEGER/BIGINT,
last_modified_at    TIMESTAMP WITHOUT TIME ZONE,
last_modified_by    INTEGER/BIGINT
```

### Fields for enum_ Tables ONLY

```sql
id                  INTEGER NOT NULL PRIMARY KEY,  -- Fixed, not auto-generated!
order_code          VARCHAR(4),
code                VARCHAR(20) NOT NULL,
short_name          VARCHAR(50) NOT NULL,
full_name           VARCHAR(100) NOT NULL,
created_at          TIMESTAMP WITHOUT TIME ZONE NOT NULL DEFAULT NOW(),
last_modified_at    TIMESTAMP WITHOUT TIME ZONE
-- NO created_by, NO last_modified_by
```

### Additional Fields by Prefix

| Prefix | state_id | organization_id | status_id + table_id | doc_on + doc_number | created_by / last_modified_by |
|--------|----------|-----------------|----------------------|---------------------|-------------------------------|
| enum_  | ✅ (some)| ❌ NO           | ❌                   | ❌                  | ❌ NO                          |
| info_  | ✅       | ❌ NO           | ❌                   | ❌                  | ✅ YES                         |
| hl_    | ✅       | ✅ **REQUIRED** | ❌                   | ❌                  | ✅ YES                         |
| doc_   | ❌       | ✅ **REQUIRED** | ✅ **REQUIRED**      | ✅ **REQUIRED**     | ✅ YES                         |
| sys_   | varies   | varies          | ❌                   | ❌                  | ✅ YES                         |

**Key Rules:**
- **`enum_` and `info_`**: Global tables, NO `organization_id`
- **`hl_` and `doc_`**: Always tenant-specific, MUST have `organization_id`
- **`sys_`**: Depends on context (global vs domain-specific)
- **`doc_` tables**: MUST have `table_id` foreign key referencing `adm.sys_table(id)`

## Field Naming Conventions

### General Rules

- **Tables:** Singular form only (`sys_user`, NOT `sys_users`)
- **Columns:** snake_case lowercase with underscores
- **Timestamps:** Use `created_at`, `last_modified_at` (NOT `date_of_created`, `date_of_modified`)
- **User references:** Use `created_by`, `last_modified_by` (NOT `created_user_id`, `modified_user_id`)

### Type Suffixes

| Suffix | Usage | Examples |
|--------|-------|----------|
| `_id` | Integer/BigInt foreign keys or IDs | `user_id`, `organization_id`, `employee_id` |
| `_at` | Timestamp/datetime fields | `created_at`, `last_modified_at`, `confirmed_at`, `cancelled_at` |
| `_on` | Date fields (date type only) | `birth_on`, `start_on`, `end_on`, `doc_on` |
| `_url` | URL fields | `image_url`, `profile_url`, `document_url` |
| `_code` | Code/reference fields | `order_code`, `code`, `cadastre_code` |

### Examples

```sql
-- Timestamp fields (TIMESTAMP WITHOUT TIME ZONE)
created_at          TIMESTAMP WITHOUT TIME ZONE
last_modified_at    TIMESTAMP WITHOUT TIME ZONE
confirmed_at        TIMESTAMP WITHOUT TIME ZONE
cancelled_at        TIMESTAMP WITHOUT TIME ZONE

-- Date fields (DATE type)
doc_on              DATE
birth_on            DATE
start_on            DATE
end_on              DATE

-- User audit fields (INTEGER or BIGINT)
created_by          INTEGER
last_modified_by    BIGINT
```

**Important Notes:**
- Use `_at` suffix for timestamps
- Use `_on` suffix for date-only fields
- Use `_by` suffix for user references (NOT `_user_id`)
- `doc_` tables use `doc_on` for the document date field

## sys_table Metadata Registration

### CRITICAL RULE

**When creating ANY new table, you MUST register it in `adm.sys_table`.**

**The `INSERT INTO adm.sys_table` statement MUST be placed in the SAME script file as the `CREATE TABLE` statement — at the end of that file.**

### sys_table Structure

```sql
CREATE TABLE adm.sys_table (
    id              INTEGER NOT NULL PRIMARY KEY,
    short_name      VARCHAR(250),
    full_name       VARCHAR(300),
    db_schema_name  VARCHAR(150),
    db_table_name   VARCHAR(150),
    table_type      VARCHAR(150)
);
```

### Registration Examples

```sql
-- Example 1: enum table
INSERT INTO adm.sys_table(id, short_name, full_name, db_schema_name, db_table_name, table_type)
VALUES (1, 'enum_gender', 'enum_gender', 'ADM', 'enum_gender', 'ENUM');

-- Example 2: info table
INSERT INTO adm.sys_table(id, short_name, full_name, db_schema_name, db_table_name, table_type)
VALUES (22, 'info_country', 'info_country', 'ADM', 'info_country', 'INFO');

-- Example 3: hl table
INSERT INTO adm.sys_table(id, short_name, full_name, db_schema_name, db_table_name, table_type)
VALUES (171, 'hl_employee', 'hl_employee', 'HRM', 'hl_employee', 'HL');

-- Example 4: doc table
INSERT INTO adm.sys_table(id, short_name, full_name, db_schema_name, db_table_name, table_type)
VALUES (632, 'doc_cadastre', 'doc_cadastre', 'ORG', 'doc_cadastre', 'DOC');

-- Example 5: translate table
INSERT INTO adm.sys_table(id, short_name, full_name, db_schema_name, db_table_name, table_type)
VALUES (23, 'info_country_translate', 'info_country_translate', 'ADM', 'info_country_translate',  'TRANSLATE');

-- Example 6: table related to doc table
INSERT INTO adm.sys_table (id, short_name, full_name, db_schema_name, db_table_name, table_type)
 VALUES (674, 'doc_music_school_regional_rate_table', 'doc_music_school_regional_rate_table', 'FIN', 'doc_music_school_regional_rate_table', 'TABLE');
```

### Registration Rules

- **Main tables:** Include `table_type` field matching the table prefix (uppercase: `ENUM`, `INFO`, `HL`, `DOC`, `SYS`)
- **Translate tables:** Include `table_type` field matching the table prefix (uppercase: `TRANSLATE`)'
- **table related to doc tables:** Include `table_type` field matching the table prefix (uppercase: `TABLE`)
- **Schema names:** Uppercase abbreviation (`ADM`, `HRM`, `ORG`, `STD`, `FIN`, `COM`, `MY`, `APP`, `LMS`)
- **ID assignment:** Choose next available ID (check existing max ID in sys_table first)

## Translation Table Pattern

### Purpose

Support multilingual content by storing translations for specific fields in separate tables.

### Naming Convention

`{main_table_name}_translate`

**Examples:**
- `info_country` → `info_country_translate`
- `enum_status` → `enum_status_translate`
- `hl_position` → `hl_position_translate`

### Required Fields

```sql
id                  INTEGER GENERATED ALWAYS AS IDENTITY,
owner_id            INTEGER                     NOT NULL,
language_id         INTEGER                     NOT NULL,
column_name         VARCHAR(100)                NOT NULL,
translate_text      VARCHAR(500)                NOT NULL,
created_at          TIMESTAMP WITHOUT TIME ZONE NOT NULL DEFAULT NOW(),
created_by          BIGINT,
last_modified_at    TIMESTAMP WITHOUT TIME ZONE,
last_modified_by    BIGINT
```

### Rules

- **NO `is_deleted`** — translate tables do not have soft delete
- **NO `ON DELETE CASCADE`** — FK to owner must NOT use CASCADE
- **UNIQUE constraint required** — `UNIQUE (owner_id, language_id, column_name)` is mandatory

### SQL Creation Example

```sql
CREATE TABLE adm.info_country_translate (
    id                  INTEGER GENERATED ALWAYS AS IDENTITY,

    owner_id            INTEGER                     NOT NULL,
    language_id         INTEGER                     NOT NULL,
    column_name         VARCHAR(100)                NOT NULL,
    translate_text      VARCHAR(500)                NOT NULL,

    created_at          TIMESTAMP WITHOUT TIME ZONE NOT NULL DEFAULT NOW(),
    created_by          BIGINT,
    last_modified_at    TIMESTAMP WITHOUT TIME ZONE,
    last_modified_by    BIGINT,

    CONSTRAINT pk_adm__info_country_translate PRIMARY KEY (id),

    CONSTRAINT uc_adm__info_country_translate__lang
        UNIQUE (owner_id, language_id, column_name),

    CONSTRAINT fk_adm__info_country_translate__owner_id
        FOREIGN KEY (owner_id) REFERENCES adm.info_country (id),

    CONSTRAINT fk_adm__info_country_translate__language_id
        FOREIGN KEY (language_id) REFERENCES adm.enum_language (id)
);

INSERT INTO adm.sys_table (id, short_name, full_name, db_schema_name, db_table_name, table_type)
VALUES (0, 'info_country_translate', 'info_country_translate', 'ADM', 'info_country_translate', 'TRANSLATE');
```

### C# Entity Example

```csharp
[Table("info_country_translate", Schema = "adm")]
public partial class CountryTranslate : TranslateEntity<CountryTranslate, TranslateColumn>
{
    [Column("language_id")]
    public int LanguageId { get; set; }

    [ForeignKey(nameof(LanguageId))]
    public virtual Language Language { get; set; }

    [Column("owner_id")]
    public int OwnerId { get; set; }

    [ForeignKey(nameof(OwnerId))]
    public virtual Country Owner { get; set; }
}
```

**Important:** Translate tables must also be registered in `adm.sys_table` (see section above).

## Schema Organization

MAD ERP uses PostgreSQL schemas to organize tables by domain:

### Domain Schemas

| Schema | Purpose | Examples |
|--------|---------|----------|
| `adm` | Administration (core system) | `enum_state`, `info_organization`, `sys_user`, `sys_role` |
| `hrm` | Human Resources Management | `hl_employee`, `hl_position`, `doc_appointment_employee` |
| `org` | Organization & Structures | `hl_building`, `doc_cadastre`, `doc_structure_passport` |
| `std` | Students Management | `hl_student`, `doc_enrollment`, `hl_grade` |
| `fin` | Finance | Financial documents, accounting |
| `com` | Commissions | Commission management |
| `my` | My Applications | User-specific workflows, `doc_application` |
| `app` | Applications & Workflows | Application processing |
| `lms` | Learning Management | Courses, curriculum |
| `link` | Integration/Linking | Cross-system data linking |
| `mtb` | Material-Technical Base | Assets, inventory |

### Schema Assignment Rules

| Table Type | Schema | Examples |
|------------|--------|----------|
| Common enum tables | `adm` | `enum_state`, `enum_status`, `enum_gender` |
| Common info tables | `adm` | `info_country`, `info_organization`, `info_currency` |
| Common system tables | `adm` | `sys_user`, `sys_role`, `sys_table` |
| HR-specific tables | `hrm` | `hrm.hl_employee`, `hrm.doc_appointment_employee` |
| Organization-specific tables | `org` | `org.hl_building`, `org.doc_cadastre` |
| Student-specific tables | `std` | `std.hl_student`, `std.doc_enrollment` |

```sql
CREATE TABLE IF NOT EXISTS adm.info_organization (
```

```sql
CREATE TABLE IF NOT EXISTS hrm.hl_employee (
```

### doc_ Table Example

```sql
CREATE TABLE org.doc_cadastre (
    id                  BIGINT GENERATED ALWAYS AS IDENTITY,
    doc_number          VARCHAR(30)                 NOT NULL,
```

## Foreign Key Constraint Rules

| Field | FK Constraint | References | Notes |
|-------|---------------|------------|-------|
| `organization_id` | ✅ YES | `adm.info_organization.id` | Tenant isolation |
| `state_id` | ✅ YES | `adm.enum_state.id` | Active/Passive state |
| `status_id` | ✅ YES | `adm.enum_status.id` | Document lifecycle |
| `table_id` | ✅ YES | `adm.sys_table.id` | Table metadata reference (doc_ tables) |
| `doc_table_id` (sys_*_manage) | ✅ YES | `adm.sys_table.id` | Source document table reference |
| `end_table_id` (sys_*_manage) | ✅ YES | `adm.sys_table.id` | Termination document table reference |
| `doc_id` / `document_id` (sys_*_manage) | ❌ NO | - | Polymorphic reference (table determined by doc_table_id) |
| `end_doc_id` / `end_document_id` (sys_*_manage) | ❌ NO | - | Polymorphic reference (table determined by end_table_id) |
| `created_by` | ❌ NO | - | Audit field only (NOT in enum_) |
| `last_modified_by` | ❌ NO | - | Audit field only (NOT in enum_) |
| `owner_id` (translate) | ✅ YES | `{main_table}.id` | Translation reference to parent |
| `language_id` (translate) | ✅ YES | `adm.enum_language.id` | - |

### Important Notes

- **User audit fields (`created_by`, `last_modified_by`):** Do NOT create foreign key constraints. These are audit-only fields.
- **Translation tables:** The `owner_id` foreign key references the parent table.
- **doc_ tables:** MUST have a foreign key from `table_id` to `adm.sys_table(id)` for metadata tracking.
- **sys_*_manage tables:**
  - `doc_table_id` and `end_table_id` MUST have foreign keys to `adm.sys_table(id)`
  - `doc_id` and `end_doc_id` should NOT have foreign keys (polymorphic references to different document tables)
- **IMPORTANT:** Do NOT use `ON DELETE CASCADE` in MAD ERP. Deletion logic is handled at the application level.

## State vs Status

### state_id - Operational State

- **Purpose:** Simple Active/Passive flag for reference data
- **Used in:** `enum_` (some), `info_`, `hl_` tables
- **References:** `adm.enum_state` table
- **Values:**
  - `1` = Active
  - `2` = Passive

### status_id - Business Lifecycle Status

- **Purpose:** Business process workflow status for documents
- **Used in:** `doc_` tables only
- **References:** `adm.enum_status` table
- **Always paired with:** `table_id` field
- **Values (from enum_status):**
  - `1` = Created
  - `2` = Approved
  - `3` = Not Approved
  - `4` = Modified
  - `5` = Deleted
  - `6` = Executing
  - `7` = Archived
  - `8` = Sent
  - `9` = Received
  - `10` = Not Received
  - `11` = Revoked
  - `12` = Refused
  - `13` = Executed
  - `14` = Completed/Delivered

**Important:** `doc_` tables use `status_id` (NOT `state_id`)

## Junction & History Tables

### Junction Tables (Many-to-Many Relationships)

**Naming Pattern:** `{parent_prefix}_{parent_name}_{child_name}`

**Example:** `hrm.hl_employee_position`

```sql
CREATE TABLE hrm.hl_employee_position (
    id                  INTEGER GENERATED BY DEFAULT AS IDENTITY,

    -- Junction fields
    employee_id         INTEGER                     NOT NULL,
    position_id         INTEGER                     NOT NULL,
    organization_id     INTEGER                     NOT NULL,

    -- Audit fields
    created_at          TIMESTAMP WITHOUT TIME ZONE NOT NULL DEFAULT NOW(),
    created_by          INTEGER,

    CONSTRAINT pk_hrm_hl_employee_position PRIMARY KEY (id),
    CONSTRAINT fk_hrm_hl_employee_position_employee_id
        FOREIGN KEY (employee_id) REFERENCES hrm.hl_employee(id),
    CONSTRAINT fk_hrm_hl_employee_position_position_id
        FOREIGN KEY (position_id) REFERENCES hrm.hl_position(id),
    CONSTRAINT fk_hrm_hl_employee_position_organization_id
        FOREIGN KEY (organization_id) REFERENCES adm.info_organization(id)
);
```

### History Tables (Change Tracking)

**Naming Pattern:** `{main_table_name}_history`

**Location:** Same schema as parent table

**Example:** `adm.hl_person_history`

```sql
CREATE TABLE adm.hl_person_history (
```

## Table Creation Checklist

When creating a new table, follow this checklist:

- ✅ **Choose correct prefix** based on table purpose (enum_, info_, hl_, doc_, sys_)
- ✅ **Include all standard fields:**
  - enum_: `id`, `order_code`, `code`, `short_name`, `full_name`, `created_at`, `last_modified_at` (NO user audit fields)
  - All others: `id`, `created_at`, `last_modified_at`, `created_by`, `last_modified_by`
- ✅ **Use correct field suffixes:**
  - `_at` for timestamp/datetime fields
  - `_on` for date fields
  - `_by` for user references
  - `_id` for foreign keys
- ✅ **Include prefix-specific required fields:**
  - info_, hl_: `state_id`
  - hl_, doc_: `organization_id` (REQUIRED for tenant isolation)
  - doc_: `status_id`, `table_id`, `doc_on`, `doc_number`
- ✅ **Verify organization_id requirement:**
  - enum_ and info_: NO organization_id
  - hl_ and doc_: MUST have organization_id
  - Domain tables in hrm/org/std/etc: MUST have organization_id
- ✅ **Use singular table naming** (`sys_user`, NOT `sys_users`)
- ✅ **Use snake_case** for all column names
- ✅ **Add proper foreign key constraints** (except for created_by/last_modified_by)
- ✅ **doc_ tables MUST have unique constraint** on `(organization_id, doc_number)`
- ✅ **Register table in adm.sys_table** with unique ID and correct table_type
- ✅ **Create translate table** if multilingual support needed (`{table_name}_translate`)
- ✅ **Register translate table in adm.sys_table** (without table_type field)
- ✅ **Use correct ID generation:**
  - enum_: Manual ID assignment (NOT auto-generated)
  - info_/hl_/sys_: `INTEGER GENERATED BY DEFAULT AS IDENTITY`
  - doc_: `BIGINT GENERATED ALWAYS AS IDENTITY` (for high-volume data)

## Quick Decision Tree

### Choosing Table Prefix

```
Creating new table? Ask:

├─ Fixed enum values known at dev time?
│  └─ enum_ (NO created_by/last_modified_by, NO organization_id)
│
├─ Global reference data (not tenant-specific)?
│  └─ info_ (NO organization_id, has state_id)
│
├─ Tenant-specific reference data?
│  └─ hl_ (MUST have organization_id + state_id)
│
├─ Business document with lifecycle/status?
│  └─ doc_ (MUST have organization_id + doc_on + doc_number + status_id + table_id)
│
└─ System infrastructure (user/role/log)?
   ├─ Global system table (adm schema)?
   │  └─ sys_ (NO organization_id, e.g., sys_user, sys_role, sys_table)
   └─ Domain-specific system table?
      └─ sys_ (MUST have organization_id)

→ Always register in adm.sys_table!
→ If in domain schema (hrm/org/std), MUST have organization_id!
```

### Choosing Field Suffix

```
Naming a field? Ask:

├─ Timestamp/datetime?
│  └─ {field_name}_at
│
├─ Date only?
│  └─ {field_name}_on
│
├─ User reference (audit)?
│  └─ {action}_by (e.g., created_by, last_modified_by)
│
├─ Foreign key/ID?
│  └─ {entity_name}_id
│
├─ URL?
│  └─ {field_name}_url
│
└─ Code/reference?
   └─ {field_name}_code OR just code
```

### Choosing ID Generation

```
What ID type should I use?

├─ enum_ table?
│  └─ INTEGER NOT NULL (manual assignment, NOT auto-generated)
│
├─ doc_ table (high-volume transactions)?
│  └─ BIGINT GENERATED ALWAYS AS IDENTITY
│
└─ info_, hl_, sys_ tables?
   └─ INTEGER GENERATED BY DEFAULT AS IDENTITY
```

## Summary

This document provides comprehensive guidance for database schema design in MAD ERP Backend. Key takeaways:

1. **⚠️ CRITICAL: organization_id requirement**
   - **MUST have organization_id:** All `hl_*`, `doc_*` tables and domain-specific tables
   - **NO organization_id:** `enum_*`, `info_*`, and global `sys_*` tables (in adm schema)
   - **Rule of thumb:** If table is in a domain schema (hrm, org, std), it MUST have `organization_id`

2. **5 table prefixes** categorize tables by purpose (enum_, info_, hl_, doc_, sys_)

3. **enum_ tables** are special: fixed IDs, NO user audit fields, NO organization_id

4. **Field naming conventions:**
   - Use `_at` for timestamps
   - Use `_on` for dates
   - Use `_by` for user references (NOT `_user_id`)

5. **adm.sys_table registration** is MANDATORY for every table

6. **Translation tables** follow `{table_name}_translate` pattern

7. **Foreign keys:**
   - Required: organization_id, state_id, status_id, table_id
   - NOT required: created_by, last_modified_by (audit-only)

8. **state_id vs status_id:**
   - `state_id`: Operational state (Active/Passive) for enum_/info_/hl_
   - `status_id`: Business lifecycle status for doc_ tables

9. **doc_ tables specifics:**
   - MUST have: status_id, table_id, organization_id, doc_on, doc_number
   - MUST have unique constraint: (organization_id, doc_number)
   - Use BIGINT for ID (high-volume transactions)

10. **Schema organization:** Each domain has its own schema (adm, hrm, org, std, fin, com, my, app, lms, link, mtb)

Follow these conventions to ensure consistency, tenant isolation, and maintainability across the MAD ERP database schema.
